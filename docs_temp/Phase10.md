# Phase 10 — 스테이지맵(진행형 난이도) (상세 구현 계획)

지금까지는 Mission 1 하나만 존재했고, 클리어하면 "STAGE CLEAR" 화면에서 멈췄다.
이번 Phase에서는 **스테이지를 클리어할 때마다 풍선 수/속도가 늘어나는 더 어려운
다음 스테이지로 자동 진행**하는 구조를 추가한다. [docs/features/mission.md](../docs/features/mission.md)는
Mission 1(튜토리얼)만 정의하고 있으므로, Mission 2 이후는 **고정 데이터를 계속
손으로 추가하는 대신 Mission 1을 템플릿 삼아 절차적으로 난이도를 스케일링**하는
방식을 택한다 (스테이지가 사실상 무한히 이어질 수 있음).

## 목표

- 스테이지 인덱스(0, 1, 2, …)에 따라 풍선 개수와 속도가 점점 증가하는 스테이지
  데이터를 생성한다. 0번(Mission 1)은 기존 튜토리얼 데이터를 그대로 사용한다.
- 스테이지를 클리어하면 점수/목숨은 그대로 유지한 채 다음(더 어려운) 스테이지로
  넘어간다.
- "다시하기"는 스테이지 진행도까지 포함해 완전히 처음부터 다시 시작한다 (기존
  Phase 9 동작과 동일하게 유지).

## 1. 스테이지 생성 (`src/game/stages/index.js`, 신규)

- `getStage(stageIndex)`:
  - `stageIndex === 0`이면 `MISSION_1`을 그대로 반환한다 (튜토리얼 원본 유지).
  - `stageIndex >= 1`이면 `MISSION_1`을 템플릿으로 삼아 다음을 적용해 새 스테이지
    데이터를 생성한다:
    - **풍선 개수**: `MISSION_1.balloons.length + stageIndex * BALLOON_COUNT_STEP`
      (예: 스테이지 2는 풍선 3개, 스테이지 3은 4개 …)
    - **풍선/페널티 풍선 속도**: 기존 `vx` 크기에 `1 + stageIndex * SPEED_STEP` 배율을
      곱한다 (부호는 좌우 번갈아 유지)
    - 이름은 `Mission ${stageIndex + 1}`
    - 장애물 배치와 제한 시간은 우선 Mission 1과 동일하게 재사용한다 (장애물 배치를
      스테이지별로 다르게 하는 것은 이후 확장 여지로 남긴다 — 지금은 "풍선 수/속도
      증가"라는 요청 범위에 집중한다).
- 관련 상수는 `constants.js`에 추가한다: `STAGE_BALLOON_COUNT_STEP = 1`,
  `STAGE_SPEED_MULTIPLIER_STEP = 0.15` (스테이지당 15%씩 빨라짐).

## 2. GameCanvas 리팩터링: 스테이지 전환 시 완전 리마운트 없이 데이터만 교체

지금까지 게임 루프/입력 리스너는 `useEffect`가 마운트될 때 한 번만 설정되고
언마운트될 때 정리되는 구조였다 (Phase 0). 스테이지 클리어 후 "다음 스테이지"로
넘어갈 때 점수/목숨을 유지하려면, **컴포넌트를 다시 마운트하지 않고 내부 데이터만
교체**해야 한다.

- 풍선/장애물/아이템/타이머/`cleared` 플래그처럼 스테이지가 바뀔 때마다 초기화되어야
  하는 값들을 `loadStage(stageIndex)`라는 함수로 묶는다. 이 함수는 `useEffect` 안에서
  클로저로 선언되어, 마운트 시 `loadStage(0)`을 한 번 호출해 초기 상태를 만든다.
- 반면 `player`(위치/목숨), `harpoonSystem`, `score`는 스테이지가 바뀌어도 **유지**되는
  값이므로 `loadStage` 밖에서 한 번만 생성한다.
- "STAGE CLEAR" 화면에 "다음 스테이지" 버튼을 추가한다 (React `status` 상태를
  활용한 기존 Phase 9 패턴 재사용). 클릭 시:
  1. `stageIndexRef.current += 1`
  2. `loadStage(stageIndexRef.current)` 호출로 풍선/장애물/아이템/타이머를 새
     스테이지 값으로 교체하고 `cleared = false`로 되돌린다.
  3. `setStatus('playing')`으로 오버레이/버튼을 숨긴다.
  4. 게임 루프 자체는 멈춘 적이 없으므로(`update`가 `cleared`일 때 조기 반환만 함)
     다음 프레임부터 자동으로 재개된다 — 별도로 `loop.start()`를 다시 호출할
     필요가 없다.
- "다시하기" 버튼은 기존처럼 `App.jsx`의 `gameKey`를 증가시켜 컴포넌트를 완전히
  새로 마운트한다 (스테이지 인덱스 0, 목숨/점수 초기화까지 포함한 완전 리셋).

## 3. HUD

- 스테이지 이름(`Mission N`)은 이미 HUD에 표시되고 있으므로(Phase 8), `loadStage`가
  교체한 스테이지 데이터의 `name` 필드를 그대로 참조하면 자동으로 갱신된다.

## 4. 완료(Done) 기준

- Mission 1을 클리어하면 "STAGE CLEAR" 화면에 "다음 스테이지" 버튼이 나타난다.
- "다음 스테이지"를 누르면 풍선 수가 늘고 속도가 빨라진 "Mission 2"가 시작되며,
  이전 스테이지에서 쌓은 점수와 남은 목숨이 그대로 유지된다.
- 계속 클리어할 때마다 Mission 3, 4, …로 난이도가 점점 올라간다.
- "다시하기"를 누르면 스테이지 진행도(인덱스)까지 포함해 Mission 1부터 완전히
  다시 시작한다.
- 게임 오버 화면에는 기존처럼 "다시하기"/"메인으로"만 표시된다 ("다음 스테이지"는
  클리어했을 때만 의미가 있다).
- 콘솔 에러 없음.

## 5. Phase 10 이후 연결

- 장애물 배치도 스테이지별로 달라지게 하려면 `getStage`의 장애물 생성 로직을
  별도로 확장한다.
- 무한 진행이 아니라 "N개 스테이지 클리어 시 엔딩"으로 마무리하고 싶어지면,
  `getStage`에 최대 인덱스를 두고 그 이후에는 "ALL CLEAR" 화면을 보여주는 것을
  검토한다.
