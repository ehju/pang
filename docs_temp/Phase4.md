# Phase 4 — 장애물 (상세 구현 계획)

[docs_temp/Phases_goal.md](Phases_goal.md)의 Phase 4 목표와
[docs/features/rules.md](../docs/features/rules.md#장애물-규칙)의 장애물 규칙을 구현한다.
Phase 3까지 구현된 풍선 물리(중력/바닥·벽 반사)와 작살 발사 위에, 스테이지 내부에
배치되는 장애물(플랫폼)을 추가하고 두 가지 규칙을 적용한다: **풍선은 장애물에
부딪히면 반사**되고, **작살은 장애물에 막혀 그 지점에서 사라진다**(장애물은 파괴
불가).

## 목표

- 장애물(사각형 플랫폼)을 화면에 배치하고 렌더링한다.
- 풍선이 장애물에 부딪히면 벽/바닥 충돌과 동일하게 반사(튕김)된다.
- 작살이 장애물에 닿으면 관통하지 못하고 그 지점에서 사라진다.
- Mission 데이터가 아직 없으므로(Phase 5), 테스트용 장애물 1~2개를 하드코딩한다
  (Phase 3의 임시 풍선 스폰과 동일한 패턴).

## 1. 장애물 엔티티 (`src/game/entities/obstacle.js`)

- 상태: `{ x, y, width, height }` — 축에 정렬된 사각형(AABB)이며, 스테이지 내에서
  움직이지 않는 정적 오브젝트다 (PRD 3.4: "지형(플랫폼, 사다리, 장애물)").
- `renderObstacle(ctx, obstacle)`: 회색 계열 사각형으로 그려 풍선/플레이어와 시각적으로
  구분한다.
- 별도의 `update`는 없다 (정적 오브젝트이므로 위치가 바뀌지 않음).

## 2. 풍선-장애물 충돌 및 반사

- 풍선은 원, 장애물은 사각형이므로 플레이어-풍선 충돌(Phase 3)과 유사한 근사를
  사용하되, 반사 방향까지 계산해야 한다.
- 근사 방식: 원의 **바운딩 박스**(반지름만큼 확장된 사각형)와 장애물 사각형이
  겹치는지 확인하고, 겹친 네 방향(좌/우/상/하) 중 침투 깊이가 가장 얕은 축을
  기준으로 밀어내고 해당 축의 속도(`vx` 또는 `vy`)를 반전시킨다.
  - 예: 위에서 떨어지다 장애물 상단에 부딪히면 `vy` 반전 + `y`를 장애물 상단 바로
    위로 클램프 (바닥 반사와 동일한 패턴).
  - 좌우에서 부딪히면 `vx` 반전 + `x`를 장애물 좌/우 바깥으로 클램프.
- 이 로직은 원의 정밀한 기하학적 충돌은 아니지만, 사각형 장애물과 풍선의 상호작용을
  자연스럽게 보이게 하는 데는 충분하다 (Phase 3의 플레이어 충돌 판정도 동일 수준의
  근사를 사용 중).
- `src/game/systems/collision.js`에 `resolveBalloonObstacleCollision(balloon, obstacle)`
  함수로 추가한다 (순수 함수, balloon을 직접 mutate).

## 3. 작살-장애물 충돌 (관통 불가)

- 작살의 현재 끝(tip) 좌표 `(harpoon.x, harpoon.y)`가 장애물의 사각형 범위 안으로
  들어오면 충돌로 판정한다 (작살 폭 `HARPOON_WIDTH`를 고려한 x 범위 겹침 + y가
  장애물의 `[obstacle.y, obstacle.y + obstacle.height]` 안에 들어왔는지).
- 충돌 시:
  - 작살의 `y`를 장애물 하단(`obstacle.y + obstacle.height`)으로 클램프해 한 프레임
    동안은 장애물 바로 아래에서 멈춘 것처럼 보이게 한다.
  - 이후 화면 상단 도달 시 제거되는 기존 로직과 동일하게, "더 이상 진행할 수 없는
    작살"로 처리해 해당 프레임 이후 제거한다 (장애물은 파괴되지 않고 그대로 유지).
- `harpoonHitsObstacle(harpoon, obstacle)` 판정 함수를 `collision.js`에 추가하고,
  `updateHarpoons`가 장애물 목록을 받아 이동 후 막힘 여부를 확인하도록 확장한다.

## 4. GameCanvas 통합

- 테스트용 장애물 목록을 Phase 3의 `createInitialBalloons()`와 같은 패턴으로
  `createInitialObstacles()`에 하드코딩한다 (예: 화면 중간 높이의 플랫폼 1개).
- `update(dt)` 순서:
  1. `updatePlayer`
  2. `updateHarpoons(harpoonSystem, player, obstacles, dt)` — 이동 후 장애물에 막힌
     작살 제거
  3. `updateBalloons` (중력/바닥·벽 반사)
  4. 풍선별로 모든 장애물에 대해 `resolveBalloonObstacleCollision` 적용
  5. 작살-풍선 충돌 → 분열 (Phase 3과 동일)
  6. 플레이어-풍선 충돌 → 목숨 감소 (Phase 3과 동일)
- 렌더링 순서: 배경 → 장애물 → 풍선 → 작살 → 플레이어 (장애물은 배경 다음, 다른
  오브젝트보다 먼저 그려 뒤에 깔리는 플랫폼처럼 보이게 한다).

## 5. 완료(Done) 기준

- 화면에 회색 장애물(플랫폼)이 표시된다.
- 풍선이 장애물의 위/아래/좌/우 어느 면에 부딪히든 자연스럽게 반사되어 튕겨나간다.
- 작살을 장애물을 향해 쏘면 장애물을 통과하지 못하고 그 지점에서 사라진다.
- 장애물 자체는 어떤 경우에도 파괴되거나 사라지지 않는다.
- 콘솔 에러 없음.

## 6. Phase 4 이후 연결

- Phase 5(스테이지 시스템)에서 하드코딩된 장애물/풍선 스폰을 Mission 데이터 기반으로
  교체한다. 장애물의 좌표/크기는 스테이지 데이터 스키마의 일부가 될 예정이다.
- 장애물 위에 플레이어가 올라설 수 있게 하는 것(발판 역할)은 PRD 3.4의 "지형(플랫폼,
  사다리)" 언급과 관련이 있으나, 현재 플레이어는 좌우 이동만 가능하도록 범위를
  한정했으므로(Phase 2) 이번 Phase에서는 다루지 않는다. 필요 시 별도 Phase로 검토한다.
